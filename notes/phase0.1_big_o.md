## ما هو Big O؟

Big O بيجاوب على سؤال واحد:

> **لو الـ input اتضاعف، بيحصل إيه للوقت أو الـ memory؟**

بيوصف **شكل النمو** — مش الوقت الفعلي. وبيشيل الـ constants والتفاصيل الصغيرة ويركز على السلوك عند الـ scale الكبير.

### مثال من الحياة — خزانة الملابس

تخيل عندك **خزانة ملابس** وعايز تلاقي تيشيرت معين.

**طريقة 1 — بدون ترتيب**: تفتح كل درج وتقلب فيه واحد واحد.
- لو عندك 10 دراجات = 10 خطوات
- لو عندك 100 درج = 100 خطوة
- ده **O(n)** — كل ما الخزانة بتكبر، الوقت بيكبر بنفس النسبة

**طريقة 2 — مرتبة أبجدياً**: بتفتح النص، بعدين النص اللي فيه حرفك، وهكذا.
- مهما كبرت الخزانة، بتوصل بسرعة
- ده **O(log n)** — في كل خطوة بتشيل نص الاحتمالات

## الـ Complexity Classes — من الأسرع للأبطأ

```
O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ) < O(n!)
```

### بالأرقام — الفرق بيبان عند الـ scale الكبير:

```
n = 10          n = 100         n = 1,000,000

O(1)      →  1 op       →  1 op       →  1 op
O(log n)  →  3 ops      →  7 ops      →  20 ops
O(n)      →  10 ops     →  100 ops    →  1,000,000 ops
O(n log n)→  30 ops     →  700 ops    →  20,000,000 ops
O(n²)     →  100 ops    →  10,000 ops →  10^12 ops
O(2ⁿ)     →  1,024 ops  →  💀          →  💀
O(n!)     →  3.6M ops   →  💀💀💀       →  ☠️
```

### بالأرقام (n = 10 و n = 20):

```
Complexity  │ n=10        │ n=20
────────────┼─────────────┼──────────────────────────
O(1)        │ 1           │ 1
O(log n)    │ 3           │ 4
O(n)        │ 10          │ 20
O(n log n)  │ 30          │ 86
O(n²)       │ 100         │ 400
O(2ⁿ)       │ 1,024       │ 1,048,576
O(n!)       │ 3,628,800   │ 2,432,902,008,176,640,000
```

### على الرسم البياني:

```
وقت التنفيذ
      |                                   O(2ⁿ) 💀
      |                              ____/
      |                         ____/ O(n²)
      |                    ____/
      |              _____/ O(n log n)
      |         ____/        O(n)
      |    ____/
      |___/__________________ O(log n)
      |_________________________ O(1)
      +----------------------------> n (حجم الـ input)
```

## شرح كل Class

### O(1) — Constant
مش فارق حجم الـ input — دايما نفس الخطوات.
```
arr[0]  // جيب العنصر الأول — خطوة واحدة مهما كان حجم الـ array
```

---

### O(log n) — Logarithmic
بتنص المشكلة في كل خطوة.

**تعريف**: log₂(n) = كام مرة تضرب 2 في نفسها عشان توصل لـ n؟
```
2¹  = 2    → log₂(2)       = 1
2²  = 4    → log₂(4)       = 2
2³  = 8    → log₂(8)       = 3
2¹⁰ = 1024 → log₂(1024)    = 10
2²⁰ ≈ مليون → log₂(مليون) ≈ 20
```

**المعنى العملي**: عندك مليون عنصر وعملت binary search → **20 خطوة بس**.

**مثال من الحياة — طية الورقة**:
```
طية 1 → نص الحجم
طية 2 → ربع الحجم
طية 3 → ثمن الحجم
...
طية 7 → مستحيل تتطوي

الورقة = n ، عدد الطيّات = log₂(n)
```

---

### O(n) — Linear
تعدي على كل عنصر مرة واحدة.
```
for كل عنصر في الـ array:
    اعمل حاجة   ← n خطوة
```

---

### O(n log n) — Linearithmic
بتعمل log n خطوة **لكل عنصر**.

**أشهر مثال**: Merge Sort
- **قسّم** الـ array لنص في كل مرة → log n مستوى
- **ادمج** وانت بترجع وبتلمس كل عنصر → n عملية في كل مستوى
- النتيجة: n × log n

```
[8,3,5,1,4,7,2,6]
      ↙        ↘
[8,3,5,1]   [4,7,2,6]       ← مستوى 1
  ↙  ↘        ↙  ↘
[8,3][5,1] [4,7][2,6]       ← مستوى 2
↙↘  ↙↘    ↙↘  ↙↘
[8][3][5][1][4][7][2][6]    ← مستوى 3 = log₂(8)

ثم ادمج بترتيب في الطريق الراجعة...
→ [1,2,3,4,5,6,7,8] ✅
```

**مهم**: مستحيل ترتب array عشوائي بشكل أسرع من O(n log n) — ده حد رياضي.

---

### O(n²) — Quadratic
Loop جوا loop — كل عنصر بتقارنه بكل عنصر تاني.

```
for كل عنصر i:           ← n
    for كل عنصر j:       ← n
        قارن i و j       ← 1

المجموع = n × n = n²
```

**مثال من الحياة**: 10 ناس في حفلة، كل واحد يسلم على الكل = 10 × 9 ≈ n²

---

### O(2ⁿ) — Exponential
كل عنصر جديد بيضاعف عدد الاحتمالات.

**مثال**: كل الـ subsets الممكنة لـ `[1, 2, 3]`
```
كل عنصر عندك قرارين: داخل أو بره
1 → ×2
2 → ×2
3 → ×2
= 2³ = 8 subsets
```

**مثال من الحياة**: بكتيريا بتتضاعف — كل دقيقة الخلية بتتقسم لاتنين.
```
دقيقة 0 → 1 خلية
دقيقة 1 → 2 خلايا
دقيقة 2 → 4 خلايا
دقيقة 3 → 8 خلايا
...
دقيقة 30 → مليار خلية 🦠
```

### ليه 2ⁿ أخطر بكتير من n²؟

**n²**: كل ما n يزيد بـ 1، الرقم يزيد بنسبة صغيرة:
```
n=10 → 100
n=11 → 121   (زاد 21)
n=12 → 144   (زاد 23)
```

**2ⁿ**: كل ما n يزيد بـ 1، **الرقم يتضاعف بالكامل**:
```
n=10 → 1,024
n=11 → 2,048   (اتضاعف!)
n=12 → 4,096   (اتضاعف تاني!)
n=13 → 8,192   (اتضاعف!)
```

**بالأرقام — المقارنة الكاملة**:
```
n = 5:   n²=25        vs  2ⁿ=32          ← متقاربين
n = 10:  n²=100       vs  2ⁿ=1,024       ← 2ⁿ أكبر 10x
n = 20:  n²=400       vs  2ⁿ=1,048,576   ← 2ⁿ أكبر 2500x
n = 50:  n²=2,500     vs  2ⁿ=10^15       ← 💀
```

---

### O(n!) — Factorial
الأبطأ على الإطلاق. كل خطوة بتضرب في رقم أكبر من السابق.

**مثال**: كل الترتيبات الممكنة لـ `[علي، محمد، سارة]`
```
المكان الأول: 3 اختيارات
المكان التاني: 2 اختيار (واحد اتاخد)
المكان التالت: 1 اختيار بس

3 × 2 × 1 = 6 = 3!
```

**الفرق بين 2ⁿ و n!**:
```
2ⁿ: المضاعف ثابت (دايما ×2)
  n=3 → 8, n=4 → 16, n=5 → 32

n!: المضاعف بيكبر مع كل خطوة
  n=3 → 6 (×3), n=4 → 24 (×4), n=5 → 120 (×5)
```
لذلك n! أبطأ من 2ⁿ.

---

## القواعد الـ 3 لـ Big O

1. **شيل الـ constants**: O(2n) → O(n)، و O(100) → O(1)
2. **شيل الـ lower-order terms**: O(n² + n) → O(n²)
3. **Worst case بشكل افتراضي** — إلا لو قيل غير كده

## Time vs Space Complexity

| النوع | السؤال |
|---|---|
| Time Complexity | كام خطوة بياخد الكود؟ |
| Space Complexity | كام memory بيحجز الكود؟ |

**Auxiliary Space**: الـ memory الإضافية اللي الكود *بتاعك* بيحجزها — مش الـ input نفسه.

```
// loop عادي — O(1) auxiliary space
for i := 0; i < n; i++ { ... }

// بتعمل array جديدة — O(n) auxiliary space
result := make([]int, n)

// recursion لـ n مستوى — O(n) space (call stack)
func factorial(n int) int {
    if n == 1 { return 1 }
    return n * factorial(n-1)
}
```

### الـ Call Stack:
```
factorial(4)   ← طبق 4 (فوق)
  factorial(3) ← طبق 3
    factorial(2) ← طبق 2
      factorial(1) ← طبق 1 (تحت/القاع)
```
4 frames في الـ peak = O(n) space.

## Best / Average / Worst Case

نفس الـ algorithm ممكن يتصرف بشكل مختلف حسب الـ input.

**مثال — Linear Search في `[3, 7, 1, 9, 4, 2, 8]`**:

| Case | Input | خطوات | Big O |
|---|---|---|---|
| Best | العنصر في أول خانة: `search(3)` | 1 | O(1) |
| Average | العنصر في النص: `search(9)` | n/2 | O(n) |
| Worst | العنصر في الآخر أو مش موجود: `search(8)` أو `search(10)` | n | O(n) |

**ليه بنتكلم عن Worst Case دايماً؟**
لأن لما بتبني system حقيقي، لازم تضمن الـ performance في أسوأ سيناريو. المستخدم مش هيتقبل إن التطبيق يبطأ أحياناً.

## أسئلة الجلسة + الإجابات

---

### سؤال 1 — ما هو O(n log n)؟

**السؤال**: O(n log n) معناه إيه بالظبط؟

**الإجابة**: يعني بتعمل **log n خطوة لكل عنصر**.
- أشهر مثال: Merge Sort
- في كل مستوى من الـ merge بتلمس n عنصر
- وعندك log n مستوى
- إذن: n × log n

**لماذا O(n log n) مهم**: مستحيل رياضياً ترتب array عشوائي بشكل أسرع منه.

---

### سؤال 2 — ما هو الـ log وإيه الفرق بين 2ⁿ و n²؟

**السؤال**: what is log and how 2^n is bigger than n^2?

**الإجابة على الـ log**:
- log₂(n) بتسألك: "2 مضروبة في نفسها كام مرة عشان توصل لـ n؟"
- مثال: log₂(8) = 3 لأن 2³ = 8
- المعنى العملي: مليون عنصر في binary search = 20 خطوة بس

**الإجابة على ليه 2ⁿ أكبر من n²**:
- n² بيزيد ببطء نسبي (كل خطوة بتضيف عدد ثابت تقريباً)
- 2ⁿ بيتضاعف في كل خطوة — المضاعف ثابت (×2)
- راجع جدول المقارنة في قسم O(2ⁿ) أعلاه

---

### سؤال 3 — O(n!) و مثال على O(2ⁿ)

**السؤال**: please explain O(n!) and give an example of O(2ⁿ) too

**O(2ⁿ) — مثال Subsets**:
```
[1, 2, 3] → كل المجموعات الممكنة:
[], [1], [2], [3], [1,2], [1,3], [2,3], [1,2,3]
= 8 = 2³

كل عنصر عندك قرارين: داخل أو بره → 2 × 2 × 2 = 2ⁿ
```

**O(n!) — مثال Permutations**:
```
[علي، محمد، سارة] → كل الترتيبات:
علي-محمد-سارة / علي-سارة-محمد /
محمد-علي-سارة / محمد-سارة-علي /
سارة-علي-محمد / سارة-محمد-علي
= 6 = 3!

المكان الأول: 3 اختيارات × المكان التاني: 2 × المكان التالت: 1 = 3!
```

**الفرق الجوهري**:
- 2ⁿ: المضاعف ثابت دايماً (×2)
- n!: المضاعف بيكبر في كل خطوة (×3، ×4، ×5...)
- لذلك n! أبطأ وأخطر من 2ⁿ

### تمرين 1 — Nested Loop

**السؤال**: For the nested loop (loop inside a loop), if n=1000, how many times does `do_something()` run? What is the Big O class?

**إجابتك**: 1,000,000 و O(n²) ✅ **صح تماماً**

**الشرح**: 1000 × 1000 = 1,000,000 = n²

---

### تمرين 2 — Two Separate Loops

**السؤال**: For the two separate loops (one after the other), if n=1000, how many times does `do_something()` run? What is the Big O class, and why do we drop the constant?

**إجابتك**: 2000 و O(n) because n+n=n ⚠️ **الرقم صح، الـ reasoning فيه تصحيح بسيط**

**التصحيح**: n + n ≠ n رياضياً. الصح إن n + n = **2n** — وبنقول O(n) لأننا **بنشيل الـ constant (2)**، مش لأنهم يساووا بعض. الـ 2 مش بتأثر على شكل النمو — سواء 2n أو 100n، كلهم خط مستقيم = linear.

---

### تمرين 3 — Best / Average / Worst Case

**السؤال**: Analyze linear search in `[3, 7, 1, 9, 4, 2, 8]` — all three cases.

**إجابتك**:
- Best: `search(3)` → O(1) ✅
- Worst: `search(8)` أو `search(10)` → O(n) ✅
- Average: `search(8)` أو `search(9)` → O(n/2) = O(n) ⚠️

**التصحيح**: `search(8)` ده **worst case** مش average — لأن 8 في آخر الـ array. الـ average case هو عنصر موجود في **النص** زي `search(9)` (عند index 3). `search(9)` هو المثال الصح للـ average.
